# Teamwork Contract

Project Name: TBD

### Team Member:

Darwin Zhang, Marco Wang, Tianhao Cao, Yusen Huang

### 1. Work Distribution & Roles

-   **Equitable Distribution:** Work will be broke down into different issues, team members voluntarily take the issue that they are interested in, fair work will be monitored by team member together.

-   **The Project Manager: Yusen Huang**

-   **Code Review:** Whenever someone uploads the code or documentation, he should request the person who is directly influenced by his work to do the code review, if no one is directly impacted, whoever first sees the Pull Request is going to conduct the code review.

### 2. Time & Availability

-   **Core Hours:** Everyone is expected to be online and responsive on Slack/Discord between **2:00 PM – 8:00 PM PST during weekday**, and between **2:00 PM - 11:59 PM on due day.**

-   **Off-grid:** Evenings (after 10:00 PM) and Sunday (Except when it's due day) are "Dark Zones" (no responses expected) to prevent burnout.

-   **Code Uploads:** Code must be pushed to a branch before 8:00 PM everyday so that the team member has enough time to complete code Review.

### 3. Online Collaboration Plan

-   **Stand-up:** Written summary in the group chat at **10:00 AM from Tuesday to Due day**.

    -   *Format:* 1) What I did yesterday; 2) What I’m doing today; 3) Any blockers.

-   **Weekly Milestone Plan Meeting:** Monday at 5:00 PM via Zoom for Milestone Review/Planning.

-   **Communication Channel:** Slacks

-   **Online meeting tool:** Zoom (Link and invitation will be sent out by Project Manager at the start of the milestone)

-   **Documentation:** Documentation is done by team member and will be stored in Github Repo. What to document: Weekly Meeting Agenda and minutes, daily stand-up content, updating all necessary logs.

### 4. Style of Working

-   **Sprint Deadlines:** We operate in 1-week milestone. All "Definition of Done" criteria must be met by Due day at 11:59 PM.

-   **Hackathons:** We will hold a 3-hour "Co-working/Pair Programming" session every Due day afternoon to tackle complex bugs together.

### 5. Quality & Definition of Done (DoD)

Every piece of work must meet these standards:

-   Passes all tests.

-   Passes a peer code review.

-   Documentation is updated.

-   **Code of Conduct:** We prioritize "Psychological Safety." No question is too "basic." Passive-aggressive comments in code reviews are unacceptable.
